# Troubleshooting And Practical Tips

This guide focuses on the most common issues Home Assistant users may encounter.

## Training Server Is Not Ready

If `Training Server Connection` does not become ready:

1. confirm `NILM Training Server` is installed
2. confirm it is started
3. return to `Appliance Training Session`
4. select the available training server option
5. click `Save`

Remember:

- detection is not enough by itself
- the server must be selected and saved before NILM uses it

## Upload Failed

If sending a training job fails:

- check that the training server is running
- check that the connection card shows the server as ready
- try a shorter training interval
- retry the job

Large training jobs can be heavy because they include many embeddings and targets.

## The New Model Does Not Look Good In Preview

If the dashboard preview looks weak:

1. preview the appliance on the same interval used for training
2. inspect the tooltip values:
   - power
   - ON/OFF state
   - probability `p`
   - threshold `thr`
3. check whether the appliance activity is clear in the selected range

This is often the fastest way to understand whether the model is:

- too conservative
- too noisy
- using a threshold that is too strict

## Ground-Truth Sensor Intervals Look Strange

If sensor-derived intervals do not match your intuition:

- remember that the raw sensor line and the final ON intervals are not the same object
- the displayed intervals are generated by the backend labeling logic used for training
- short OFF gaps may be bridged before short ON runs are removed

This is intentional and helps bursty appliances avoid being labeled as always OFF.

## Dashboard Uses More Memory With Disaggregate All

This is expected.

`Disaggregate All` is heavier than a single-appliance preview because it:

- runs more appliance models
- creates more prediction series
- returns a larger result payload

If you want lighter operation:

- preview one appliance at a time when debugging
- use `Disaggregate All` mainly for overview
- use shorter history ranges

## Training Uses More Memory Than Live Inference

This is normal.

Training-related tasks are heavier than live NILM operation because they involve:

- history loading
- window preparation
- embedding extraction
- label generation
- model training

The project already isolates several heavy tasks into subprocess workers to improve memory recovery, but peak memory still depends on:

- history size
- number of windows
- number of appliances

## Browser Or UI Looks Stale After A Big UI Change

Home Assistant can cache frontend assets aggressively.

If the UI still shows outdated behavior after an update:

- reload the page
- try a hard refresh in the browser
- close and reopen the app page
- if needed, clear the Home Assistant site data in the browser

This is especially relevant after renaming or reorganizing frontend components.

## Live Binary Sensor Seems Inconsistent

The live binary appliance sensor uses:

- the model’s live on/off probability
- the deployed on/off threshold stored with the model

You can inspect both from entity attributes in Home Assistant.

This makes it possible to debug whether an appliance is OFF because:

- the predicted probability is low
- or the saved threshold is strict

## Practical Tips For Better Results

- Start with one appliance that has a distinctive power signature.
- Use a mains sensor with consistent history.
- Prefer sensor supervision when a dedicated appliance sensor exists.
- Use `Disaggregate` before `Disaggregate All` when debugging.
- Compare predictions on the same interval used for training.
- Enable live publishing only for the models you really want in Home Assistant.
